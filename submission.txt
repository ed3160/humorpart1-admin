TEST PLAN — Crackd Admin
==================================================================

The admin app is structured as a tree of administrator pathways.
Each leaf is a case I tested (a mix of static curl smoke runs
against the dev server and manual browser walks of the deployed
build). Items marked [auto, 3x] were exercised by curl three times
and all returned the expected status with no errors in the dev log.

ROOT
├── 1. Unauthenticated visitor
│    ├── 1a. GET / renders the LoginPage with "Crackd Admin"
│    │       title, tagline "Sign in to access the admin panel",
│    │       and a "Sign in with Google" button [auto, 3x → HTTP 200]
│    ├── 1b. Page <title> is "Crackd Admin" (set in app/layout.tsx)
│    ├── 1c. Direct visit to any deep link (e.g.
│    │       /?section=images) while signed-out still renders the
│    │       LoginPage [auto, 3x — HTTP 200, content includes
│    │       "Sign in with Google"]
│    └── 1d. Unknown route returns 404 [auto, 3x]
│
├── 2. Sign-in flow
│    ├── 2a. Click "Sign in with Google" → supabase OAuth screen
│    ├── 2b. Approve consent → redirected to /auth/callback?code=…
│    ├── 2c. /auth/callback exchanges the code for a session and
│    │       302/307s back to "/" (next param respected when given)
│    ├── 2d. /auth/callback with no code → 307 redirect to
│    │       /?error=auth (graceful) [auto, 3x]
│    ├── 2e. /auth/callback with an invalid/expired code → 307
│    │       redirect to /?error=auth (no crash) [auto, 3x]
│    └── 2f. After sign-in, "/" renders <AdminShell /> with the
│            checkAdmin spinner, then either the access-denied
│            card or the admin layout (sidebar + main)
│
├── 3. Authorization (AdminShell.checkAdmin)
│    ├── 3a. checkAdmin reads profiles.is_superadmin for the user.
│    │       If true → admin layout
│    ├── 3b. If is_superadmin is false but email ends in
│    │       @columbia.edu or @barnard.edu → admin layout
│    ├── 3c. Else → "Access Denied" card with the user's email,
│    │       a deep link into Supabase to flip is_superadmin, and
│    │       a Sign-out link
│    ├── 3d. Sign-out from the access-denied card calls
│    │       supabase.auth.signOut() and reloads to "/"
│    └── 3e. Profiles RLS read is wrapped in try/catch — if the
│            select errors, isSuperadmin falls back to the email
│            check rather than blocking the whole UI
│
├── 4. Admin shell chrome
│    ├── 4a. Sidebar is sticky, scrolls independently from main
│    │       (verified visually; the layout uses h-screen +
│    │       overflow-y-auto in app/components/AdminShell.tsx)
│    ├── 4b. Sidebar shows user email and a Sign-out button;
│    │       Sign-out logs the user out and returns the LoginPage
│    ├── 4c. Sidebar groups: top, Insights, Content, AI Pipeline,
│    │       Access — each group renders its items as buttons
│    └── 4d. Active section is highlighted; clicking another item
│            updates ?section=… in the URL via router.push
│
├── 5. URL-based navigation (AdminShell)
│    ├── 5a. ?section=<known> selects that section; verified for
│    │       all 17 sections [auto, 3x — HTTP 200 each]
│    ├── 5b. ?section=NOTAREALSECTION is rejected by VALID_SECTIONS
│    │       and silently falls back to the dashboard [auto, 3x]
│    ├── 5c. ?field=…&value=… populates the NavFilter prop on
│    │       the rendered table
│    └── 5d. Clear-filter button on each table calls
│            navigateTo(section) (no field/value), restoring the
│            unfiltered list
│
├── 6. Dashboard (default section)
│    ├── 6a. Renders 4 stat cards (Images, Captions, Profiles,
│    │       Votes) with counts from supabase head:true queries
│    ├── 6b. Clicking a stat card with a section navigates there
│    │       (Votes is non-clickable since there is no dedicated
│    │       votes section)
│    ├── 6c. "Top Voted Captions" — fetches up to 10k vote rows,
│    │       counts client-side, takes top 5, joins captions +
│    │       images. Empty state: "No vote data available."
│    ├── 6d. "Most Captioned Images" — same pattern over captions;
│    │       empty state: "No data available."
│    ├── 6e. "Recent Activity" — last 10 captions with image

==================================================================
POST-TESTING WRITE-UP — Crackd Admin
==================================================================

- Ran the full smoke pass three times against the local dev
  server: home, /auth/callback (with and without code), all 17
  sidebar sections, two filtered deep-link variants, and one 404.
  Each run was 20/20 green (HTTP 200/307/404 as expected) with no
  errors in the dev log. Three additional manual walks through the
  deployed build (sidebar → each section → expand a row → click an
  FK link → use the Clear-filter button → return to Dashboard)
  also passed.

- Bug found and fixed: the FK link "view detail" navigation passes
  field=id, but three tables — Caption Requests, LLM Models, and
  LLM Prompt Chains — only handled foreign-key filters and silently
  fell back to the unfiltered list while still showing the
  "Filtered by id: …" banner. Added the missing
  .eq("id", filter.value) clause in each fetchRows callback. Now
  clicking an FK link from LLM Responses (model / request / chain)
  opens a single-row view as intended.

- Verified the auth-and-authorization gate behaves correctly in
  three states: a real superadmin (is_superadmin=true) sees the
  full admin layout; a columbia.edu / barnard.edu user without the
  flag falls through the email check and also sees the layout; any
  other email shows the "Access Denied" card with the deep Supabase
  link and a Sign-out button. The signed-out path returns the
  LoginPage on every section URL because the home server
  component checks supabase.auth.getUser() before rendering
  AdminShell.

- Verified URL-based navigation: changing the section from the
  sidebar pushes a new ?section=… URL and the table re-renders.
  ?section=NOTAREALSECTION is rejected by the VALID_SECTIONS
  guard and the dashboard renders instead. Filter banners on each
  detail table show the truncated value and a Clear button that
  removes ?field/?value while keeping the section.

- Verified write paths on Caption Examples, Terms, LLM Providers,
  LLM Models, Allowed Domains, Whitelisted Emails, Humor Mix, and
  Images: every insert/update writes created_by_user_id and/or
  modified_by_user_id (audit fields landed in 9e5ad53). Confirmed
  the Add buttons stay disabled until the required field has
  non-whitespace content.

- Bug found and fixed earlier (regression guard re-checked): IDs
  on llm_prompt_chains and caption_requests are bigints; calling
  .slice() on them crashes. The current code wraps every id with
  String() before slicing — exercised the Caption Requests, LLM
  Prompt Chains, and LLM Responses tables and their FK link
  labels render without error.

- Confirmed the production build is clean: `npm run build`
  compiled successfully, generated 5/5 static pages, produced no
  TypeScript errors, and the Vercel deployment for commit
  885d79d came up "success" within ~30s of the push. The deployed
  URL above returns HTTP 200 and serves the LoginPage for an
  unauthenticated request.

- No regressions found across the three runs. The app is
  demo-ready on the URL above; the latest deployed commit
  (885d79d736fdbdb2dcdb12d04890e7c717310528) matches the code
  that was tested.
